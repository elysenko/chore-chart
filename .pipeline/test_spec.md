## API tests

### `POST /api/auth/signup`
- **Happy path**: Body `{name, email:"new@ex.com", password:"pw12345", avatarColor:"#ff0"}` on an empty DB → `200/201` with `{token, member}`; `member` includes `id, name, email, avatarColor, points:0, role:"ADMIN"` and **no** `passwordHash`. First-ever signup is flagged `ADMIN`.
- **Happy path (second user)**: A signup when at least one member already exists → member `role:"USER"`.
- **Validation failures**: Missing `email` or `password` → `400`; malformed email → `400`; empty body → `400`.
- **Duplicate email**: Signup with an email already in the DB → `409` (or `400`), no second row created.
- **Idempotency / edge cases**: Returned token must verify via `GET /api/me`; `passwordHash` never appears in any response.

### `POST /api/auth/login`
- **Happy path**: Body `{email, password}` from a seeded credential → `200` with `{token, member}` (safe fields only, no `passwordHash`).
- **Validation failures**: Missing `email` or `password` → `400`.
- **Auth failures**: Wrong password → `401`; unknown email → `401`. Error message must not distinguish "no such user" from "bad password".
- **Idempotency / edge cases**: Token returned is accepted by guarded routes as `Authorization: Bearer <token>`.

### `POST /api/auth/logout`
- **Happy path**: Any request → `200` (stateless no-op). Token remains technically valid (stateless JWT); client discards it.
- **Edge cases**: Works with or without a token present (no server session state).

### `GET /api/me`
- **Happy path**: With a valid Bearer token → `200` with the current member's safe fields incl. `role`; no `passwordHash`.
- **Auth failures**: No `Authorization` header → `401`; malformed/garbage token → `401`; expired/tampered token → `401`.

### `GET /api/members`
- **Happy path**: Authenticated → `200` with an array of ≥4 members after seed; each item has `id, name, avatarColor, points`.
- **Auth failures**: No token → `401`.
- **Edge cases**: `passwordHash` and `email` not required, but `passwordHash` must never be present.

### `GET /api/members/:id`
- **Happy path**: Authenticated, valid `id` → `200` with profile (`id, name, avatarColor, points, role`) plus `history`: their assignments joined to chores, ordered by week.
- **Validation failures**: Non-numeric `id` → `400`; unknown numeric `id` → `404`.
- **Auth failures**: No token → `401`.

### `GET /api/assignments?week=`
- **Happy path (default week)**: Authenticated, no `week` param → `200` for the current ISO week; calls `ensureWeekAssignments`; returns one entry per chore with `chore` fields, `assignee` (member), `dueDay`, `completed`. Count equals number of seeded chores (≥6).
- **Happy path (explicit week)**: `?week=2026-W29` → `200` returning that week's rotation; materializes rows if absent.
- **Rotation correctness**: For two adjacent weeks (`W_n`, `W_{n+1}`), the assignee for a given chore differs (rotation advances by one member; verify formula `(rotationOffset + weekIndex) % memberCount`).
- **Validation failures**: `?week=garbage` or malformed ISO week (e.g. `2026-W54`) → `400`.
- **Auth failures**: No token → `401`.
- **Idempotency / edge cases**: Repeated calls for the same week return identical assignment ids (no duplicate rows; guarded by `@@unique([choreId, isoWeek])`).

### `POST /api/assignments/:id/complete`
- **Happy path**: Called by the member who is the assignee of an incomplete assignment → `200`; assignment `completed:true`, `completedAt` set; that member's `points` increased by the chore's `pointValue`.
- **Idempotency**: Second call on an already-completed assignment → succeeds/no-op with **no** additional points (no double-count); `completedAt` unchanged.
- **Auth failures**: No token → `401`. Authenticated member who is **not** the assignee → `403`, assignment unchanged, no points awarded.
- **Validation failures**: Non-numeric or unknown `id` → `404`.
- **Data integrity**: Completion runs in a transaction — assignment flip and points increment either both apply or neither.

### `GET /api/leaderboard`
- **Happy path**: Authenticated → `200` with members ordered by `points` descending. After a completion that raises a member's points above a peer, that member ranks higher.
- **Auth failures**: No token → `401`.
- **Edge cases**: Ties are returned in a stable order (e.g. by id); no `passwordHash` present.

### `GET /api/admin/settings`
- **Happy path (admin)**: Admin token → `200` listing provisioned service keys (postgresql, minio) with **masked** values and a `configured` boolean per key.
- **Auth failures**: No token → `401`; authenticated non-admin (`USER`) → `403`.
- **Edge cases**: Raw secret values are never returned unmasked.

### `PATCH /api/admin/settings`
- **Happy path (admin)**: Admin token, body of key/value pairs → `200`; values upserted into `SystemSetting`; subsequent `GET` shows `configured:true` for those keys.
- **Auth failures**: No token → `401`; non-admin → `403`, no write performed.
- **Validation failures**: Malformed body (not key/value object) → `400`.

### `GET /api/health`
- **Happy path**: Unauthenticated → `200` with `{status:'ok'}`.
- **Auth**: Must succeed with **no** token (public).

### `GET /api/health/deep`
- **Happy path**: Unauthenticated → `200` after a trivial DB query succeeds.
- **Auth**: Public (no token required).
- **Edge cases**: If the DB is unreachable, returns a non-200 (e.g. `503`) rather than throwing.

## UI / journey tests

### Journey: Login with seeded credential
- **Steps**: Visit `/login`; enter a seeded email + password from `SEED_CREDS_JSON`; submit.
- **Expected outcomes**: Redirect to `/` (Dashboard); JWT persisted in `localStorage`; nav shows the member and a Logout control.
- **Negative path**: Wrong password → stays on `/login`, shows an error message, no token stored.

### Journey: Guarded route redirect when unauthenticated
- **Steps**: With no token in `localStorage`, navigate directly to `/` (or `/members`, `/leaderboard`).
- **Expected outcomes**: `RequireAuth` redirects to `/login`; protected content not rendered.
- **Negative path**: Underlying guarded API call returns `401`; SPA does not crash.

### Journey: Signup (first user becomes admin)
- **Steps**: On an empty DB, visit `/signup`; fill name/email/password/avatar; submit.
- **Expected outcomes**: Authenticated and redirected to `/`; member `role` is `ADMIN`; Admin link visible in nav.
- **Negative path**: Duplicate email or missing field → inline error, remains on `/signup`.

### Journey: Dashboard "This Week"
- **Steps**: Logged in, visit `/`.
- **Expected outcomes**: Renders a **"This Week"** heading; lists each chore → assignee → due day (weekday name); populated with assignments after seed.
- **Negative path**: API error → error state shown; empty week → empty state, not a crash.

### Journey: Dashboard week navigation via `?week=`
- **Steps**: Visit `/?week=2026-W29`, then `/?week=2026-W30`.
- **Expected outcomes**: Each renders that ISO week's rotation (read via `useSearchParams`); assignee for a given chore differs between the two adjacent weeks (rotation advances). Deep-link is addressable directly (no click-through).
- **Negative path**: Invalid `week` value → error/empty state.

### Journey: Mark own chore complete
- **Steps**: On `/`, find a chore assigned to the logged-in member that is incomplete; click "Mark complete".
- **Expected outcomes**: Chore flips to done (button removed/disabled); after refetch the member's points increase; leaderboard reflects it.
- **Negative path**: "Mark complete" button is **not** shown for chores assigned to other members; a repeat completion does not add points again.

### Journey: Members grid
- **Steps**: Logged in, visit `/members`.
- **Expected outcomes**: Grid lists ≥4 members, each with name, avatar color swatch, and points.
- **Negative path**: Loading and empty states render without error.

### Journey: Member detail deep-link
- **Steps**: Navigate directly to `/members/:id` (no click-through from the grid).
- **Expected outcomes**: Renders the member's profile + chore history, route-addressable.
- **Negative path**: Unknown id → not-found / error state, not a crash.

### Journey: Admin settings access control
- **Steps**: As `ADMIN`, open `/admin/settings`; view service (postgresql, minio) status; submit a credential value. Then as a `USER`, attempt to open `/admin/settings`.
- **Expected outcomes**: Admin sees per-service configured/unconfigured badges and a credential form; PATCH persists and flips the badge to configured. Non-admin does not see the Admin link and is blocked/redirected from the page.
- **Negative path**: Non-admin PATCH attempt is rejected (`403`); no settings changed.

## Data integrity tests
- After `POST /assignments/:id/complete`: exactly one `Assignment` row is `completed:true` with a non-null `completedAt`; the assignee's `Member.points` rose by exactly the chore's `pointValue` (never twice for the same assignment).
- `Assignment` obeys `@@unique([choreId, isoWeek])`: `ensureWeekAssignments` never creates duplicate rows for the same chore+week across repeated calls.
- `Member.email` is unique: signup with an existing email creates no second row.
- Rotation determinism: for a fixed member set ordered by `id`, `ensureWeekAssignments(week)` yields the same assignee mapping on every call for that week.
- Points are monotonic under completions and never negative; leaderboard order equals `points` descending at all times.
- Seed idempotency: running the seed twice (upserts) does not duplicate members or chores.

## Out of scope
- **Real postgresql/minio integration** — the spec declares `Integrations: None`; the admin settings scaffold stores values in `SystemSetting` but no runtime behavior consumes them, so only the settings CRUD + access control is tested (per the open question in `tasks.md`).
- **Token expiry duration / refresh tokens** — spec does not define TTL or refresh; only accept/reject of valid vs invalid tokens is tested.
- **Password strength / complexity rules** — spec is silent; only presence/format validation is tested.
- **Rate limiting, CSRF, CORS specifics** — not described in the spec.
- **Concurrency of simultaneous completions on the same assignment** — beyond the documented single-request idempotency guarantee.
- **ISO week 53 / year-boundary exact values** — flagged as a risk to unit-check in `week.ts`; correctness of the helper is a server unit concern rather than an API/journey case here (rotation-advance behavior is covered).
- **SQLite volume persistence across container restarts** — deployment/infra concern, not application behavior.
